Project Overview
Add a Client Dashboard to the existing Applier Dashboard system. This is a separate user experience accessed via unique client login credentials. Clients will view their job search progress, approve application materials, access interview prep documents, and track applications.

System Architecture
User Roles

Admin (Wilson) - Existing applier dashboard + new client management capabilities
Client - New user type with separate dashboard experience
Applier - Existing user type (no changes)

Authentication

Separate login page for clients (different from applier login)
Admin creates client accounts via "Add New Client" button
Auto-generated credentials (username/password) provided to client
Session timeout after 30 minutes of inactivity
Logout button in client dashboard
No password reset functionality (Wilson manages passwords)


Client Dashboard - Four Main Panels
1. Overview Panel
Layout:
Three clickable pill-shaped buttons/cards displayed horizontally:
Pill 1: Applications Sent

Click to expand and show:

Total applications sent (all time)
Weekly applications sent
Daily applications sent


Collapsed state shows only total number

Pill 2: Rejections

Single number display (total rejections)
No expansion/drill-down

Pill 3: Interviews

Single number display (total interviews scheduled)
No expansion/drill-down

Data Updates:

Real-time or semi-real-time updates as admin adds data
If real-time is too slow, acceptable to refresh every 30-60 seconds


2. Interviews Panel
Calendar Display:

Dark mode Google Calendar-style interface
Shows only interviews added by admin (read-only for client)
Month/week/day view options
Each interview event is clickable

Interview Event Details:
When client clicks an interview event, open modal/page showing:
Header:
Company: [Company Name]
Role: [Job Title]
Interview Date/Time: [Formatted DateTime]
Interview Format: [Phone/Video/Panel/In-Person]
Interviewer(s): [Names + Titles]
Job Description: [Clickable link]
Section 1: Company Overview

What they do (2-3 sentences)
Company size
Funding/growth trajectory
Culture vibe

Section 2: Recent News & Updates

Product launches, acquisitions, changes (last 90 days)
Earnings highlights if applicable
Pain points/challenges

Section 3: What This Role Actually Does

Day-to-day responsibilities
Team structure
Success metrics (first 90 days)
Key skills needed

Section 4: About Your Interviewer(s)

Name, title, background
LinkedIn activity highlights
Personal interests/hobbies
Rapport builders (2-3 conversation hooks)

3-5 tailored questions based on company research

What do people who succeed here have in common?

Italicized text: "We'll send a tailored thank-you email on your behalf after the interview for follow-up."

Client Interaction:

Clients can highlight text and add comments (like document approval flow)
Comments saved and visible to admin
Client cannot edit/rewrite content, only comment

Mobile Responsive:

Must be viewable on mobile devices
PDF export option available

Google Calendar Integration:

Admin adds interview → automatically pushes to client's Google Calendar
Client authorizes calendar access during onboarding
One-way sync (admin to client calendar only)


3. Applying UI Panel
Top Navigation:
Three pill-shaped tabs: Resume | Cover Letters | LinkedIn

Resume Tab
Initial State (Before Approval):
Left Side (70% of screen):

Display "old resume" (uploaded by admin)
Multi-page PDF viewer with page navigation
Takes up most of the vertical space

Right Side (30% of screen):

"Improve" button (prominent, styled)

Interaction:

Client clicks "Improve" button
Old resume spins quickly (animation)
Flips/pops with flash effect to reveal new resume
New resume displays in same viewer

After Flip:

Bottom of screen:

Approve button (green)
Deny button (red)


Client can highlight portions of new resume
Highlighting triggers lightbulb icon → click to add comment
Comments saved and visible to admin

Approval Action:

Click Approve → locks in resume
Small celebration animation
Auto-advances to Cover Letters tab

Denial Action:

Click Deny → saves comments
Document marked as "Needs Revision" for admin
Client can still view but cannot re-approve until admin uploads new version

Post-Approval State:

Only new resume visible (old resume removed)
No Approve/Deny buttons
No ability to add new comments
Read-only view


Cover Letters Tab
Similar flow to Resume, with additions:
A/B/C Testing Pills:

After flip animation, show pills above document: A | B | C
Each pill represents a different cover letter version uploaded by admin
Click pill to instantly switch between versions (fast loading, no animation)
Client can view differences by toggling between versions

Approval:

Single Approve All button at bottom (approves A, B, AND C together)
Client doesn't pick one—all versions get approved
Same highlight/comment functionality per version
After approval → locks all versions, advances to LinkedIn tab

Post-Approval:

Can still toggle between A/B/C to view
No comments or approve/deny options


LinkedIn Tab
Same structure as Cover Letters:

Old → New flip animation
A/B/C pills for multiple LinkedIn PDF versions
Highlight and comment functionality
Approve All button (approves all versions)
After approval → locks and completes document review process

Final State After All Approvals:

All three tabs (Resume, Cover Letters, LinkedIn) show approved versions
Read-only mode
Client can navigate between tabs and versions but cannot edit/comment

Display:

List view of all applications sent for this specific client
Same tile/card design as Applier Dashboard job listings

Each Application Tile Shows:

Company name
Job title
Date applied

Search & Filter:

Search bar at top: filter by job title or company name
Sortable columns:

Date (newest/oldest)
Company (A-Z)
Status (group by status type)



Pagination:

Load 20 applications initially
"Load More" button at bottom
Clicking loads next 20 (infinite scroll style)

Status Updates:

Admin manually updates status from admin panel
Status changes reflect in real-time (or semi-real-time) on client dashboard

Technical Specifications
Tech Stack

Use same stack as existing Applier Dashboard
Frontend: React (or existing framework)
Backend: Node.js / Express (or existing)
Database: PostgreSQL / MongoDB (or existing)
Authentication: JWT or existing auth system

Calendar Library

Replit's choice (suggested: FullCalendar.js or react-big-calendar)
Must support dark mode styling

Document Viewer

PDF viewer library with page navigation
Must support highlighting and commenting (suggested: react-pdf with annotation layer)

Animation Library

For resume flip effect: Replit's choice
Should be smooth, fast (< 1 second total animation time)
Celebration animation on approval: confetti or similar

Google Calendar API Integration

OAuth 2.0 for client calendar access
One-way sync: admin adds event → pushes to client's Google Calendar
Include event details: title, date/time, description with prep doc link

File Upload

Support PDF, .doc, .docx
No file size restrictions (admin uploads only)
Store in cloud storage (S3, Cloudinary, or existing system)

Real-Time Updates

WebSocket connection for live data updates (if feasible)
Fallback: polling every 30 seconds
Updates needed for: applications count, rejections, interviews, new documents

Comments System

Store comments tied to:

Document ID (resume/cover letter/LinkedIn)
Client ID
Highlighted text selection (start/end position)
Timestamp


Display in admin panel with context (show highlighted section + comment)

Notifications for Admin

Red badge indicator on client card when:

Document denied with comments
New comments added to prep doc


Badge clears when admin views comments
